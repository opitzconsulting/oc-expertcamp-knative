Change to OpenShift Workshop Project

The following checks ensure that each chapter exercises are done with the right environment settings.

OpenShift

  • OpenShift CLI should be v4.2

oc version

The output should be like

Client Version: v4.2.2
Kubernetes Version: v1.14.6+2e5ed54

  • Make sure to be on knative{number} OpenShift project

oc project -q

If you are not on knative{number} project, then run following command to change to knative{number} project:

oc project knative{number}

The {number} has to be replaced by the number assigned to you.

Deploy the Service

Navigate to the tutorial chapter’s serving folder:

#Linux
cd $TUTORIAL_HOME/src/main/knative/serving
#Windows
cd %TUTORIAL_HOME%/src/main/knative/serving
oc -n knative{number}  apply -f service-v1.yaml

The following snippet shows what a Knative service YAML looks like:

service-v1.yaml

apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: helloworld
spec:
  template:
    metadata:
      name: helloworld-v1
      annotations:
        # disable istio-proxy injection
        sidecar.istio.io/inject: "false"
    spec:
      containers:
      - image: quay.io/rhdevelopers/knative-tutorial-greeter:quarkus
        livenessProbe:
          httpGet:
            path: /healthz
        readinessProbe:
          httpGet:
            path: /healthz

After successful deployment of the service we should see a Kubernetes Deployment named similar to helloworld-nsrbr-deployment available:

See what you have deployed

The service-based deployment strategy that we did now will create many Knative resources, the following commands will help you to query and find what has been deployed.

service

oc --namespace knative{number}  get services.serving.knative.dev helloworld

configuration

oc --namespace knative{number} get configurations.serving.knative.dev helloworld

routes

oc --namespace knative{number} get routes.serving.knative.dev helloworld

revisions

oc --namespace knative{number} get rev --selector=serving.knative.dev/service=helloworld --sort-by="{.metadata.creationTimestamp}"

  • add -oyaml to the commands above to see more details

Deploy a New Revision of a Service

As Knative follows 12-Factor application principles, any configuration change will trigger creation of a new revision of the deployment.

To deploy a new revision of the helloworld service, we will add an environment variable to the existing service as shown below:

Service revision 2

service-env.yaml

apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: helloworld
spec:
  template:
    metadata:
      name: helloworld-v2
      annotations:
        # disable istio-proxy injection
        sidecar.istio.io/inject: "false"
    spec:
      containers:
      #- image: docker.io/fliegera/helloworld:v1
      - image: quay.io/rhdevelopers/knative-tutorial-greeter:quarkus
        env:
        - name: MESSAGE_PREFIX
          value: Namaste
        livenessProbe:
          httpGet:
            path: /healthz
        readinessProbe:
          httpGet:
            path: /healthz

Adding an environment variable that will be used as the message prefix

Let us deploy the new revision using the command: :service-file: service-v1.yaml

oc -n knative{number}  apply -f service-env.yaml

After successful deployment of the service we should see a Kubernetes deployment called helloworld-v2-deployment.

Now if you list revisions, you will see two of them, named similar to helloworld-v1 and helloworld-v2.

revisions

oc --namespace knative{number} get rev --selector=serving.knative.dev/service=helloworld --sort-by="{.metadata.creationTimestamp}"

Invoking Service will now show an output like Namaste helloworld ⇒ '6fee83923a9f' : 1, where Namaste is the value we configured via environment variable in the Knative service resource file.

Pinning Service to a Revision

As you noticed, Knative service always routes traffic to the latest revision of the service deployment. It is possible to split the traffic amongst the available revisions.

As we already know the at got two revisions namely helloworld-v1 and helloworld-v2. The traffic block in the Knative service specification helps in pinning a service to a particular revision or split traffic among multiple revisions.

You can use the show Knative service revisions command to find the available revisions for the helloworld service.

apiVersion: serving.knative.dev/v1alpha1
kind: Service
metadata:
  name: helloworld
spec:
  template:
    metadata:
      name: helloworld-v2
      annotations:
        # disable istio-proxy injection
        sidecar.istio.io/inject: "false"
    spec:
      containers:
      #- image: docker.io/fliegera/helloworld:v1
      #- image: image-registry.openshift-image-registry.svc:5000/knative01/meetup-greeting:1
      - image: quay.io/rhdevelopers/knative-tutorial-greeter:quarkus
        env:
        - name: MESSAGE_PREFIX
          value: Namaste
        livenessProbe:
          httpGet:
            path: /healthz
        readinessProbe:
          httpGet:
            path: /healthz
  traffic:
  - tag: current
    revisionName: helloworld-v1
    percent: 100
  - tag: prev
    revisionName: helloworld-v2
    percent: 0
  - tag: latest
    latestRevision: true
    percent: 0

The above service definition creates three sub-routes(named after traffic tags) to existing helloworld route.

  • current - The revision is going to have all 100% traffic distribution

  • prev - The previously active revision, which will now have zero traffic

  • latest - The route pointing to any latest service deployment, by setting to zero we are making sure the latest revision is not picked up automatically.

Let us redeploy the helloworld service by pinning it to the helloworld-v1:

oc -n knative{number}  apply -f service-v1-pinned.yaml

Let us list the available sub-routes:

oc -n knative{number} get ksvc helloworld -oyaml

After executing this command you should find in the output the following helloworld routes:

- http://current-helloworld.knative01.apps.meetup.openshift.opitz.cloud (1)
- http://prev-helloworld.knative01.apps.meetup.openshift.opitz.cloud (2)
- http://latest-helloworld.knative01.apps.meetup.openshift.opitz.cloud (3)
1 the sub route for the traffic tag current
2 the sub route for the traffic tag prev
3 the sub route for the traffic tag latest

Invoking Service will produce output similar to Hi helloworld ⇒ '6fee83923a9f' : 1

As per the current traffic distribution, the helloworld route will always return response from revision helloworld-v1 (traffic tag current)

  • If you are on OpenShift then you can directly invoke the urls to see the response of those respective revision
